# Review action harness

> **Workflow Status**: Requirements gathered ✓ → Ready for `/create_plan_generic` to research and plan

## Problem Statement & Business Context

**What problem are we solving?**
After running `review-analyzer`, we have a set of VALID findings with proposed fixes in markdown files. Currently, implementing these fixes is a manual process. We need an automated harness that reads review-analyzer output and implements each fix sequentially.

**Why is this important?**
- Eliminates manual effort to apply review feedback
- Creates a repeatable, auditable pipeline: analyze → fix → validate → commit
- Enables automated review-to-fix workflows in CI/CD

**Who needs this?**
- Developers using the OCR → review-analyzer → fix pipeline
- CI/CD pipelines that want to auto-remediate code quality issues

## Requirements & Scope

### Must-Have Features
- Read review-analyzer output directory (INDEX.md if present, else iterate all markdown finding files)
- Parse per-finding markdown files to extract: file path, line range, existing code, suggested code, review comment, verdict, analysis
- Filter to only VALID verdict items
- Abstract coding agent interface (`CodingAgent` protocol/ABC) with default opencode SDK implementation
- Agent abstraction must support swapping to Claude SDK (or other providers) without changing harness logic
- Sequential processing: for each VALID item → analyze fix → apply code changes → run lint/typecheck → commit → next
- One atomic git commit per fix with descriptive message (include finding ID, file, brief description)
- Post-fix validation: run `ruff check` + `ruff typecheck` (or `mypy`) on affected file
- Skip item (log warning, continue to next) if validation fails
- Final summary report: items processed, commits made, skipped items, errors

### Nice-To-Have Features
- Configurable validation commands (allow custom lint/typecheck commands per project)
- Dry-run mode: show what would be done without making changes
- Parallel processing of independent findings (different files)
- JSON summary output in addition to human-readable summary
- Resume capability: continue from last processed item if interrupted

### Out of Scope
- Re-running review-analyzer (harness consumes its output)
- Handling REJECTED or BACKLOG items
- Interactive approval per fix
- Non-Python language support initially

## Constraints & Context

**Integration Points**:
- `review-analyzer` output directory (default: `feedback/`)
- Git repository (commits to current branch)
- opencode CLI via subprocess (default agent)
- Linting: `ruff check`, `mypy` (or project's configured linters)

**Preferred Technologies**:
- Python, matching existing codebase patterns
- opencode SDK as default coding agent
- `claude-agent-sdk` as alternate agent implementation
- No new heavy dependencies; reuse existing packages

**Constraints**:
- Must work within existing project structure (`deep_architect/` package)
- Must follow existing coding conventions (ruff, mypy)
- Agent abstraction must be clean enough that swapping providers is a config change, not a refactor
- Each commit must be atomic and reversible

## Research Questions for Planning Phase

The following should be investigated during `/create_plan_generic`:
- How should the coding agent abstraction be designed to support both opencode (subprocess) and Claude SDK (direct API)?
- What is the exact structure of the per-finding markdown files and what parsing approach is most robust?
- How should the commit message format incorporate finding details?
- What is the best approach for post-fix validation — subprocess calls to ruff/mypy, or programmatic APIs?
- Should the harness live as a new module (`deep_architect/review_action_harness.py`) or under a subpackage?
- How does the existing `deep_architect/agents/client.py` Claude SDK integration inform the agent abstraction?
- How should errors during code application be handled — retry, skip, or abort?
- What configuration options should be exposed via CLI args and `~/.deep-architect.toml`?
