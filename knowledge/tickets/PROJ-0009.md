# Support Creating Architecture from Code Base

> **Workflow Status**: Requirements gathered ✓ → Ready for `/create_plan` to research and plan

## Problem Statement & Business Context

**What problem are we solving?**
`adversarial-architect` currently only supports greenfield projects driven by a BMAD PRD. There is no way to generate C4 architecture documentation for an *existing* codebase.

**Why is this important?**
Senior engineers, architects, consultants, and AI agents frequently need to create or recover architecture documentation for existing systems. This adds a reverse-engineering mode so `adversarial-architect` can serve that use case.

**Who needs this?**
Any senior engineer, architect, consultant, or AI agent working on an existing codebase. This is a general-purpose addition to the `adversarial-architect` CLI.

## Reference Implementation (To Research)

**Location**: existing greenfield flow in this repository (`deep_architect/harness.py`, `deep_architect/agents/`, `deep_architect/prompts/`)

**Aspects to mimic**:
- Same 7-sprint structure and sprint topics (Context → Container → Component → Code → cross-cutting concerns → security → ADRs)
- Same output format: Mermaid + Markdown C4 artifacts
- Same shared infrastructure: harness, exit criteria, git ops, models, `load_prompt` mechanism
- Differ only in: CLI entry point, prompts, and sprint definitions

> Note: Detailed analysis of reference will happen during planning phase

## Requirements & Scope

### Must-Have Features
- New `--reverse-engineer` CLI flag (alongside existing `--greenfield`)
- New `--codebase <path>` flag specifying the target git repository to analyze
- Generator prompts rewritten for code-reading (no PRD input); Generator reads the target codebase via its file tools (Read, Glob, Grep, Bash)
- Critic prompts rewritten for accuracy verification — Critic confirms the generated docs accurately reflect the actual code (not adversarial critique)
- Critic remains read-only (Read, Glob, Grep, Bash — no Write/Edit)
- Output (`knowledge/architecture/`) written relative to the target codebase directory (not the CWD)
- Git commits for output files made in the target codebase repo
- All 7 sprint topics unchanged from greenfield
- README updated with `--reverse-engineer` / `--codebase` usage

### Nice-to-Have Features
- None identified at this time

### Out of Scope
- No changes to sprint count (stays at 7)
- No new output formats beyond existing Mermaid + Markdown
- No LLM or agent framework changes
- Greenfield mode completely unchanged
- No support for non-git directories (git repo required — needed for commits)

## Constraints & Context

**Integration Points**:
- Target must be a git repository (validated at startup, same as greenfield's `validate_git_repo()`)
- Output path is `<codebase>/knowledge/architecture/` (relative to the analyzed repo)
- Git commits for generated files go into the target repo

**Preferred Technologies**: Python, existing `deep_architect` package patterns

**Constraints**:
- Must not break existing `--greenfield` mode or its tests
- New prompts must follow the existing `.md` file convention loaded via `load_prompt()`
- No hardcoded thresholds — all from `HarnessConfig.thresholds`

## Success Criteria

### Automated Verification
- [ ] `adversarial-architect --reverse-engineer --codebase <path>` runs the full 7-sprint loop without error
- [ ] Generator reads target codebase files via its tools (not prompt-injected content)
- [ ] Critic reads target codebase files via its tools (read-only, no Write/Edit)
- [ ] Output artifacts land in `<codebase>/knowledge/architecture/`
- [ ] Git commits are made in the target codebase repo
- [ ] All existing greenfield unit tests continue to pass
- [ ] New reverse-engineer prompt `.md` files exist and are covered by `test_prompts.py`
- [ ] `uv run ruff check`, `uv run mypy`, `uv run pytest`, `uv run bandit` all pass clean

### Manual Verification
- [ ] Run `--reverse-engineer` against this (`deep-architect`) codebase and review the generated C4 docs for accuracy

**End State**: `adversarial-architect --reverse-engineer --codebase /path/to/repo` produces a complete, accurate C4 architecture document set in `<repo>/knowledge/architecture/` via the same adversarial loop, with no regressions in greenfield mode.

## Research Questions for Planning Phase

The following should be investigated during `/create_plan`:

- How does `harness.py` currently wire the PRD path into generator prompts — what's the cleanest way to swap in a codebase path instead?
- Should `--greenfield` / `--reverse-engineer` be a mutually exclusive flag group or a positional `mode` argument?
- What changes are needed in `cli.py` to support the new flags and route to the right harness entry point?
- Are separate `SprintDefinition` lists needed (one for greenfield, one for reverse-engineer), or can sprint definitions be mode-parameterized?
- What new prompt files are needed (generator system, critic system, sprint 1–7 variants) — can any be shared across modes?
- How should `generator-history.md` and `generator-learnings.md` be scoped when the output dir is in the target repo?
- Does `init_workspace()` in `files.py` need changes to support a non-CWD output root?
- Should `test_prompts.py` `EXPECTED_PROMPTS` grow a separate list for reverse-engineer prompts?
