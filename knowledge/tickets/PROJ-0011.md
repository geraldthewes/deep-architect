# review-analyzer tool

> **Workflow Status**: Requirements gathered ✓ → Ready for `/create_plan_generic` to research and plan

## Problem Statement & Business Context

**What problem are we solving?**
OCR (Open Code Review, https://github.com/alibaba/open-code-review) generates a large volume of code review findings in JSON format. Many of these findings are false positives, low-value, or not actionable. There's no tool to systematically evaluate each finding's validity before acting on them.

**Why is this important?**
- A single OCR run on `plant-tracking` produced **242 comments + 30 warnings across 84 files**
- Manually triaging 272 review items is impractical
- Using a second LLM (different model) provides an independent perspective to validate/critique each finding
- Filtering by path is needed to exclude agent tool configs that shouldn't be changed

**Who needs this?**
Developers using OCR for code review who need to triage findings before implementing fixes.

## Reference Implementation (To Research)

**Location**: `.opencode/commands/analyze-review-feedback.md`

**Aspects to mimic**:
- The analysis prompt structure (confirm issue, explain, critique, suggest alternatives)

## Requirements & Scope

### Must-Have Features
- CLI tool (`review-analyzer`) that accepts OCR JSON file path as input
- Per-item LLM analysis via `opencode run` (non-interactive, programmatic)
- Path filtering via `--include` / `--exclude` CLI flags (glob patterns)
- One markdown output file per review item in a `feedback/` directory
- Output filename: `{filepath_hash}-{item_index}.md` (filepath hash + item index for uniqueness when multiple items share a path)
- Output file contains: original OCR finding + LLM's analysis/critique/verdict
- Configurable LLM model via `--model` CLI flag
- Configurable concurrency via `--concurrency` CLI flag
- Summary report at the end (counts by verdict category, e.g., valid, rejected, backlog)

### Nice-to-Have Features
- Resumable runs (skip already-analyzed items on re-run)
- JSON output format option for downstream tooling
- Integration with deep-architect's ticket/workflow system later

### Out of Scope
- Auto-applying fixes (that's for a later, more adversarial tool)
- Interactive UI / TUI
- Managing the `opencode serve` process

## Constraints & Context

**Integration Points**:
- Uses `opencode run` CLI for LLM calls (subprocess)
- Reads OCR JSON output format from https://github.com/alibaba/open-code-review
- Lives in the `deep-architect` repo as a standalone CLI tool

**Preferred Technologies**:
- Python (consistent with repo)
- `opencode run` with `--format json` for machine-parseable LLM output

**Constraints**:
- Must support concurrent LLM calls (rate-limited by API)
- Must handle LLM failures gracefully (circuit breaker pattern exists in this repo)
- Must work non-interactively (no user prompts during batch processing)
