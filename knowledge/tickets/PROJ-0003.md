# Save context at each round for generator and critic and start new session

> **Workflow Status**: Requirements gathered ✓ → Ready for `/create_plan` to research and plan

## Problem Statement & Business Context

**What problem are we solving?**
The generator and critic agents currently reuse the same `session_id` across all ping-pong rounds within a sprint. Accumulated context grows over long runs and causes failures, degradation, and unpredictable behavior.

**Why is this important?**
Long-run context accumulation is a known source of instability. Moving to a file-mediated, stateless-session model eliminates this root cause. It also makes `--resume` the normal execution path rather than an untested edge case — resume and a fresh start become the same mechanism.

**Who needs this?**
Any use of the harness on non-trivial runs (i.e., all real usage). This is a correctness and reliability fix as much as a feature.

## Requirements & Scope

### Must-Have Features

- Each generator turn starts a **new session** — no `session_id` reuse across rounds
- Each critic turn starts a **new session** — no `session_id` reuse across rounds
- At the start of each turn, the agent loads all required context exclusively from files (sprint contract, prior critic feedback, sprint prompt, etc.) — not from prior session memory
- Generator appends a structured history entry per round to `generator_history.md`
- Critic appends a structured history entry per round to `critic_history.md`
- History file format: **Markdown with strong section labels/keys** (grep-friendly); entries include what was decided/changed (generator) or what concern was raised and its resolution status (critic)
- Both agents are provided the path to their history file and can search it via `Grep`/`Read` tools; the file is **not** auto-loaded into context
- `--resume` works through the same file-loading path with no special-case logic — resume is the normal flow

### Nice-to-Have Features

- Critic history entries cross-reference the generator round that addressed (or failed to address) a concern
- History entries are searchable by sprint number, round number, or concern keyword

### Out of Scope

- Changing the 7-sprint structure or sprint definitions
- Changing the critic's scoring / exit criteria logic
- Persistent session memory across runs (history is per-project workspace, not global)

## Constraints & Context

**Integration Points**: `claude-agent-sdk` session management, `io/files.py` save/load layer, `harness.py` orchestration loop

**Preferred Technologies**: Markdown for history files (grep-friendly); existing Pydantic models for any structured data

**Constraints**:
- `session_id` is scoped to a single turn — it must not be threaded across rounds
- History files must be written to the project workspace (same location as contracts, feedback, progress files)
- Filesystem size is not a concern — append-only history is fine
- No agent framework changes — plain async Python only

**End State**: Generator and critic each operate as stateless-session agents that communicate exclusively through files. History files accumulate a searchable record of decisions and concerns across all rounds. Resume is the default mechanism, not a special path.

## Research Questions for Planning Phase

1. How is `session_id` currently threaded through `run_generator()` and `run_critic()`? What exactly needs to change to scope it to one turn?
2. What does `generator_learnings.md` currently store and how is it written? Can it become (or seed) the generator's round history, or does it serve a distinct purpose?
3. What context is currently injected into generator/critic prompts vs. discovered via tools? Which pieces need to become explicit file-loads at session start?
4. What is the right file naming/location convention for history files — per-sprint or global across all sprints for the workspace?
5. Are there any session-resumption guarantees in `claude-agent-sdk` that the harness currently relies on that would break when `session_id` is no longer reused?

## Notes

The core insight is that file-mediated handoff between turns is strictly more robust than session-context continuity for long multi-round runs. The history files give agents the ability to recall prior decisions without bloating the active context window.
