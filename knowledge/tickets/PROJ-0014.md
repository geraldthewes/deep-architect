# Add support for grok coding agent to review-action

> **Workflow Status**: Requirements gathered âś“ â†’ Ready for `/create_plan_generic` to research and plan

## Problem Statement & Business Context

**What problem are we solving?**
`review-action` currently supports two coding agent backends for its fix loop: **Claude** (via the Claude Agent SDK) and **opencode**. This ticket adds a **third backend, Grok Build (xAI)**, so users have another model choice for cost/speed/preference reasons.

**Why is this important?**
Gives users more control over which LLM backend runs the fix loop â€” `review-action` is already multi-backend (Claude + opencode), and Grok is a natural third option when it's cheaper, faster, or otherwise preferred for a given run.

**Who needs this?**
Users of the `review-action` CLI who want to choose their coding agent/model backend per run â€” extending the existing Claude/opencode choice with a Grok option.

## Reference Implementation (To Research)

**Location**: `deep_architect/agents/client.py`

**Aspects to mimic**:
- The existing `run_agent` / `run_agent_structured` abstraction in `client.py` used to drive Claude via the Claude Agent SDK
- **The existing opencode integration is the closer, concrete reference** for adding a third subprocess-based backend. Located in `deep_architect/review_action_harness.py`:
  - An `OpencodeAgent` class that shells out to a subprocess binary (`self.opencode_bin`, default path, overridable via `OPENCODE_BIN` env var) and parses its NDJSON stdout (`_parse_opencode_ndjson`) to detect success
  - A `--provider` CLI flag (currently `choices=["opencode", "claude"]`, default `"opencode"`, around line 1460) and a `provider: str = "opencode"` field on the run config (around line 88) that should be extended with a `"grok"` choice
  - How Grok is added should follow this same shape (a `GrokAgent`-equivalent class, subprocess or SDK call, extended `--provider` choices) unless research finds a strong reason to diverge

> Note: Detailed analysis of `client.py`, the full `OpencodeAgent` implementation (fix-apply and quality-check-fix code paths, NDJSON parsing) as the precedent for adding a third backend, and how Grok's SDK/API/CLI maps onto this shape, will happen during the planning phase.

### Rough Grok Build Notes (Unverified â€” researcher must independently verify during planning)

The user supplied the following rough, unverified pointer info on **Grok Build**, xAI's terminal-native coding agent CLI. Treat all of this as a starting lead, not fact â€” the planning-phase researcher must independently verify current accuracy (versions, flags, pricing, auth flow) against official docs before relying on it:

- CLI install/run: `grok -p "<prompt>"` â€” headless/non-interactive mode intended for scripts, automation, CI/CD, background agents. Runs with full codebase awareness from the current directory.
- Useful flags (unverified): `--output-format streaming-json` (structured streaming output for scripts), `-m <model-name>` (e.g. `grok-build-0.1` or `grok-4.5`).
- Auth: browser login on first run; in headless/non-browser environments, set `XAI_API_KEY` env var â€” this would align with this project's env-var-only auth convention.
- Also has ACP (Agent Client Protocol) support â€” JSON-RPC over stdio/HTTP â€” for running Grok as a backend agent in other harnesses/editors.
- Alternative integration path: direct xAI API + `xai-sdk` (Python) or `@ai-sdk/xai`/OpenAI-compatible client (JS/TS), model `grok-build-0.1` (256K context, ~$1/M input, $2/M output) or `grok-4.5`, giving full tool-calling/multi-turn/structured-output control instead of shelling out to the CLI.
- Claimed official docs to check: `https://docs.x.ai/build/overview` (headless usage), `https://docs.x.ai/developers/quickstart` (API/SDK), `https://x.ai/cli` (install).
- Claimed access requirement: Grok Build CLI beta requires a SuperGrok or X Premium+ subscription; direct API access uses `XAI_API_KEY` from the xAI console.

This raises an open design question for planning: integrate via the `grok` CLI in headless `-p` mode (shelling out, parsing streaming-json), or integrate via the `xai-sdk`/API directly (more control, avoids CLI subscription gating)? Both should be weighed against how well each maps onto the `run_agent`/`run_agent_structured` interface in `client.py`.
