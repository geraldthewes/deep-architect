# Adding support for resuming an architecture design

> **Workflow Status**: Requirements gathered ✓ → Ready for `/create_plan` to research and plan

## Problem Statement & Business Context

**What problem are we solving?**
Long architecture runs (up to 3 hours, 7 sprints, 40 rounds) can be interrupted mid-run by process kills, LLM endpoint failures, hard timeouts, or power loss. Without checkpointing, the entire run must restart from scratch.

**Why is this important?**
The main drivers are **time** (not re-paying for completed LLM rounds) and **correctness** (preserving already-agreed architecture artifacts from completed sprints). Re-running completed sprints risks producing different outputs and wastes significant LLM compute.

**Who needs this?**
Both solo developers running locally and headless/CI contexts where unattended runs must be resumable without human intervention.

## Scope & Requirements

### Must-Have Features (MVP)
- Checkpoint written to disk after every successful Generator pass (sprint N, round M complete)
- On `--resume`, detect the checkpoint, skip completed sprints/rounds, and continue from the last incomplete round
- Checkpoint captures: current sprint index, round number, scores so far, file paths written
- If no checkpoint exists and `--resume` is passed, fail fast with a clear error message
- Checkpoint written atomically (write-to-tmp then `os.replace`) — no partial/corrupt writes on kill
- Checkpoints stored in a dedicated `.checkpoints/` directory at repo root
- `.checkpoints/` added to `.gitignore` (not for long-term storage — easy to wipe)
- Already-written architecture files trusted as correct on resume (no re-validation)

### Nice-to-Have Features
- `--resume` auto-detects the most recent checkpoint in `.checkpoints/` without requiring a path argument
- Human-readable checkpoint format (JSON) so a developer can inspect or manually edit it

### Out of Scope
- Resuming mid-round (if the Generator is killed mid-tool-call, that round restarts from the beginning)
- Any UI or progress dashboard
- Checkpoint retention or archiving

## Constraints & Context

**Integration Points**:
- `HarnessProgress` / `SprintStatus` models in `models/progress.py` — may already capture sufficient state; needs verification
- `--resume` CLI flag in `cli.py` — exists as a stub; needs wiring into `harness.py`
- `io/files.py` — checkpoint save/load to be added here if not already present
- `.gitignore` — `.checkpoints/` must be excluded

**Preferred Technologies**:
- Python `os.replace()` for atomic writes
- JSON for checkpoint format (reliable, human-readable)
- Existing `pydantic` models for serialization

**Constraints**:
- Checkpoints must NOT land in `knowledge/architecture/`, `logs/`, or any directory containing data to be kept
- `.checkpoints/` at repo root is the designated location — isolated, obvious to delete

## Success Criteria

### Automated Verification
- [ ] `--resume` with no checkpoint present fails fast with a clear, descriptive error
- [ ] Killing the process mid-sprint and re-running with `--resume` continues from the correct sprint/round (no completed work re-run)
- [ ] Checkpoint file is written atomically (no partial/corrupt state on SIGKILL)
- [ ] `.checkpoints/` is present in `.gitignore`
- [ ] All existing tests still pass; new unit tests cover checkpoint save/load round-trip

### Manual Verification
- [ ] Kill a live run after sprint 2 completes; `--resume` picks up at sprint 3 with architecture files from sprints 1–2 intact
- [ ] Deleting `.checkpoints/` and re-running without `--resume` starts fresh with no errors

**End State**: An interrupted run can be restarted with `adversarial-architect --prd knowledge/prd.md --output knowledge/architecture --resume` and continue exactly where it left off, with no completed sprint re-executed.

## Research Questions for Planning Phase

The following should be investigated during `/create_plan`:
1. What does `HarnessProgress`/`SprintStatus` in `models/progress.py` already capture — is it sufficient for resume, or does it need new fields (e.g., round-level granularity)?
2. Is the `--resume` CLI flag in `cli.py` already wired into `harness.py`, or is it a stub with no implementation?
3. Where in `harness.py` should checkpoint writes be inserted — after Generator pass, after Critic pass, or both?
4. Does `io/files.py` already have any checkpoint save/load logic, or does it need to be added from scratch?
5. Are there existing unit tests for `HarnessProgress` that can be extended for resume scenarios?
6. What is the correct atomic write pattern for checkpoint files in Python (`os.replace` vs `tempfile`)?

## Notes

- The `.checkpoints/` directory is intentionally transient — it should be safe to delete at any time between runs. It is not part of the architecture output and should not be committed.
- Mid-round resume (replaying individual tool calls) is explicitly out of scope — the atomic unit of recovery is a completed Generator pass.
