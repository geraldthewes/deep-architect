# Keep-Best Rollback on Score Regression

> **Workflow Status**: Requirements gathered ✓ → Ready for `/create_plan` to research and plan

## Problem Statement & Business Context

**What problem are we solving?**
The adversarial generator ↔ critic loop currently logs "regressed" when a generator round scores lower than the previous round, but takes no corrective action. The next round proceeds from the regressed files, potentially compounding the regression over multiple rounds.

**Why is this important?**
Without rollback, a bad generation round can permanently damage the architecture baseline, causing subsequent rounds to improve from a degraded state rather than the best-known state. This means the harness may produce a final architecture that is worse than it was at some intermediate round — wasted compute and lower output quality.

**Who needs this?**
All users of the `adversarial-architect` CLI. This is an improvement to the core optimization loop that benefits every run.

**Reference inspiration**: Karpathy's AutoResearch uses a hill-climbing with rollback strategy — if a generation round scores lower than the best-so-far, discard the new files and start the next round from the best-scoring baseline.

## Requirements & Scope

### Must-Have Features

1. **`restore_arch_files_from_commit(repo, best_commit_sha) -> list[str]`** in `git_ops.py`
   - Diffs best_commit → HEAD to find files changed after the best round
   - Restores Modified/Deleted files via `repo.git.checkout(best_commit_sha, '--', path)` (stages to index + working tree)
   - Deletes Added-after-best files from disk and removes from index
   - Handles Renamed files (delete renamed copy, restore original)
   - Excludes `generator-learnings.md`, `generator-history.md`, `critic-history.md` from restore (accumulated knowledge survives)
   - Returns list of relative paths affected (empty = nothing to revert)

2. **`git_commit_staged(repo, message) -> bool`** in `git_ops.py`
   - Commits whatever is currently staged in the index
   - Detects staged changes via `repo.index.diff("HEAD")`
   - Returns True if a commit was created, False if nothing staged
   - Needed because `repo.git.checkout(sha, '--', path)` stages directly to index, bypassing the normal `get_modified_files()` / `index.diff(None)` path

3. **`append_rollback_event(output_dir, sprint_num, round_num, regressed_score, best_score, best_commit_sha)`** in `io/files.py`
   - Appends a `[ROLLBACK]`-prefixed entry to `generator-history.md`
   - Entry format includes action, reason, reverted commit SHA, and guidance for the generator

4. **Harness integration** in `harness.py`:
   - Track `best_result: CriticResult | None` and `best_commit_sha: str | None` per sprint
   - On resume: seed `best_result` by scanning prior round feedback (leave `best_commit_sha = None` — rollback only safe with new commits from this session)
   - After each round: update best if current score exceeds best
   - Rollback block fires when `result.average_score < best_result.average_score - 0.05` (0.05 tolerance to avoid noise)
   - Rollback placed **after** exit criteria and ping-pong checks (a passing round never rolls back)
   - `last_result` set to `best_result` after rollback (next generator sees best-version feedback)

5. **5 new unit tests** in `tests/test_git_ops.py`:
   - `test_restore_arch_files_restores_modified` — modified arch file is restored
   - `test_restore_arch_files_skips_excluded` — `generator-learnings.md` not restored even when changed
   - `test_restore_arch_files_deletes_added_files` — files added after best are deleted on restore
   - `test_restore_arch_files_noop_when_at_best` — returns `[]` when HEAD == best commit
   - `test_git_commit_staged_commits_staged` / `_noop_when_nothing_staged` — helper creates commit iff index has staged changes

### Nice-to-Have Features
- None identified; scope is well-defined

### Out of Scope
- Changing rollback threshold (0.05) to be configurable via `HarnessConfig` — hardcoded is fine for now
- Cross-sprint rollback (rollback state resets at each sprint boundary)
- Rollback of `critic-history.md` (critic history always accumulates)

## Constraints & Context

**Integration Points**:
- `git_ops.py` — new functions sit alongside existing `git_commit()`, `get_modified_files()`
- `io/files.py` — new function appends to `generator-history.md` (existing file managed by harness)
- `harness.py` — sprint loop modified to track best and trigger rollback
- `tests/test_git_ops.py` — 5 new tests using real temp git repos (consistent with existing pattern)

**Key ordering constraint**: `best_result` update must happen **before** the rollback check so the current round's score is captured as best before the next round can trigger rollback.

**`consecutive_passes` is NOT reset on rollback** — if the best round was a passing score, the pass count is preserved correctly.

**`generator-learnings.md` survives rollback** — the generator's notes about what failed remain visible for future rounds.

## Implementation Notes

### `restore_arch_files_from_commit` key detail
`repo.git.checkout(sha, '--', path)` stages changes **directly to the git index** (not just the working tree), so `get_modified_files()` (which checks `index.diff(None)`) won't see them. That's why `git_commit_staged` uses `repo.index.diff("HEAD")` instead.

### Rollback block placement (ordering rationale)
- Best-tracking update before rollback check → best captures round-N score; rollback fires on round-N+1
- Rollback fires after consecutive-passes and ping-pong logic → a passing round never rolls back
- `generator-learnings.md` excluded from `_EXCLUDED_FROM_ROLLBACK` so agent notes about failures remain
