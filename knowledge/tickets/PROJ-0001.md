# Implement Adversarial C4 Architecture Harness (Deep Researcher)

> **Workflow Status**: Requirements gathered ✓ → Ready for `/create_plan` to research and plan

## Problem Statement & Business Context

**What problem are we solving?**
The BMAD "Create Architecture" step currently requires manual iteration between an architect and a reviewer to harden a C4 architecture document. This is time-consuming and quality is inconsistent.

**Why is this important?**
A fully autonomous adversarial harness eliminates the manual back-and-forth. Run it once and come back to a production-ready, critique-hardened C4 architecture document — no babysitting required.

**Who needs this?**
Any developer or team using the BMAD methodology who wants high-quality architecture documents without manual iteration.

## Reference Implementation (To Research)

**Location**: `https://github.com/coleam00/adversarial-dev`

**Aspects to mimic**:
- Overall adversarial agent loop structure (Generator ↔ Critic)
- File-based communication between agents (contracts, feedback, artifacts)
- Exit/convergence criteria patterns
- How sprints/steps are structured and sequenced

> Note: Detailed analysis of reference will happen during planning phase — this is critical input for the design.

## Requirements & Scope

### Must-Have Features (MVP)
- Standalone Python package (`pip install` from repo), CLI entry point: `adversarial-architect`
- Two configurable OpenAI-compatible API endpoints: one for Generator, one for Critic (model name + API key + base URL each)
- Sprint-based C4 generation in order:
  1. C1 System Context
  2. C2 Container (overall + tech stack)
  3. Frontend container details
  4. Backend / Orchestration container
  5. Database + Knowledge Base
  6. Edge Functions / Deployment / Auth / Scaling / Observability
  7. Final integration + cross-cutting concerns (ADRs, non-functional requirements)
- Per-sprint adversarial loop: contract creation → critic tightens → generator builds → critic scores → repeat
- Exit criteria per sprint: avg score ≥ 9.0, zero Critical/High issues, 2 consecutive rounds
- Ping-pong detection: LLM-based similarity check using the Critic LLM (cosine similarity > 85% + no severity decrease → auto-exit)
- Global safety limits: max 6 rounds/sprint, 40 total rounds, 3-hour hard timeout
- Final "Mutual Agreement" round: both agents independently output `READY_TO_SHIP`
- Output exactly matches `knowledge/architecture/` folder structure (see below)
- Mermaid C4 diagrams (C1, C2, optional C3)
- File-based communication: contracts, feedback, artifacts all written to disk
- `--resume` flag to restart interrupted runs
- Auto-commit after each successful Generator pass (requires git repo — fail fast if not in one)
- Rich console logging + `logs/architect-run-YYYYMMDD-HHMMSS/` directory
- Fully reproducible: seed + JSON logs of every round
- Config loaded from `~/.deep-researcher.toml`, overridable by CLI flags

### Nice-to-Have Features
- None for MVP

### Out of Scope
- C4 Code-level (C4) diagrams or detailed component implementation
- Actual code generation
- UI or web interface — CLI only
- LangGraph fallback (deepagents only)
- Per-repo config files

### Output Folder Structure
```
knowledge/architecture/
├── c1-context.md
├── c2-container.md
├── deployment.md
├── frontend/
│   ├── c2-container.md
│   ├── auth.md
│   ├── knowledge-base.md
│   ├── pitch-generation.md
│   └── zod-schemas.md
├── database/
│   └── c2-container.md
├── edge-functions/
│   └── c2-container.md
├── contracts/
├── feedback/
└── decisions/   (ADRs)
```

## Constraints & Context

**Integration Points**:
- Any two OpenAI-compatible API endpoints (provider-agnostic)
- Git (hard requirement — must be run inside a git repo)

**Preferred Technologies**:
- Python 3.12+
- `deepagents` (LangChain Deep Agents)
- `pydantic` for contracts/feedback schemas
- `typer` or `click` for CLI
- `gitpython` for auto-commits
- `tomllib` (stdlib) for config parsing
- `rich` for console output

**Constraints**:
- Must be installable as a standalone pip package usable across multiple repos
- Config in `~/.deep-researcher.toml` (not per-repo) since it's a shared system-level tool
- Standard OpenAI Bearer token auth for API endpoints
- Hard fail if not run inside a git repo
