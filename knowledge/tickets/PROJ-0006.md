# Add adversarial-plan command

> **Workflow Status**: Requirements gathered ✓ → Ready for `/create_plan` to research and plan

## Problem Statement & Business Context

**What problem are we solving?**
The plan review step in the SDLC is currently manual and human-in-the-loop. `adversarial-architect` automates architecture hardening end-to-end, but once an implementation plan is created, critique and revision falls back to humans. This breaks the adversarial automation and is a bottleneck.

**Why is this important?**
The existing adversarial framework (generator ↔ critic loop, sprint contracts, quality gates, auto-commits, structured feedback) is already proven in `adversarial-architect`. Extending it to plan generation produces higher-quality implementation plans through adversarial pressure, multi-expert critique (via `architecture_critique.md`), and wiki-enforced best practices — without manual review bottlenecks.

**Core goal**: Produce a **well-engineered implementation plan** that follows solid software engineering principles as defined by the `software-backend-wiki` and the `create_plan_generic.md` framework. The plan is the deliverable, not code.

**Who needs this?**
Gerald, as the primary user of the `deep-architect` toolchain, orchestrating the full PRD → Architecture → Plan → Implementation pipeline.

## Reference Implementation

**Location**: `deep_architect/` — the entire `adversarial-architect` implementation is the reference.

**Key modules to reuse or extend**:
- `harness.py` — orchestration loop (round management, ping-pong detection, resume logic)
- `agents/client.py` — `make_agent_options()`, `run_agent()`, `run_agent_structured()`
- `agents/generator.py` / `agents/critic.py` — agent run patterns
- `models/contract.py`, `models/feedback.py`, `models/progress.py` — Pydantic models (extend, don't duplicate)
- `io/files.py` — save/load for contracts, feedback, progress; workspace init
- `exit_criteria.py` — `sprint_passes()`, `should_ping_pong_exit()` (reuse as-is)
- `git_ops.py` — `git_commit()`, `get_modified_files()` (reuse as-is)
- `sprints.py` — `SprintDefinition` pattern (mirror for plan sprints)
- `prompts/` — `.md` prompt loading pattern via `load_prompt()`
- `cli.py` — Typer entry point pattern (add new command here or new entrypoint)

> Note: The goal is maximum code reuse. Shared logic (exit criteria, git ops, agent client, file I/O, Pydantic models) must not be duplicated. New plan-specific modules should follow the exact same patterns and conventions as existing modules.

## Requirements & Scope

### Must-Have Features

#### CLI

- New CLI command `adversarial-plan` (separate entrypoint from `adversarial-architect`, sharing the same package)
- Accepts one or more input documents (PRD, C4 architecture docs, or any relevant context files)
- `--resume` flag restores progress after crash or interruption, same mechanism as `adversarial-architect`
- `--strict` mode (fail hard on max-round exhaustion vs. soft-fail with best-effort output)

#### 8-Sprint Structure

All 8 sprints are treated equally — same contract negotiation, quality gates, and generator/critic loop applies to every sprint including Sprint 1.

| Sprint | Name | Focus |
|--------|------|-------|
| 1 | Wiki Scoping | Identify which sections of `software-backend-wiki` are applicable; agree on scope contract; create plan document skeleton |
| 2 | Requirements Analysis | Task decomposition, functional requirements, acceptance criteria per task |
| 3 | Dependency Ordering | Sequencing, dependency graph, critical path identification |
| 4 | Test Strategy | Unit/integration/e2e testing approach, acceptance criteria per phase |
| 5 | Risk Assessment | Edge cases, failure modes, mitigations |
| 6 | Architecture Critique | Critique using `architecture_critique.md` framework; per-issue research/challenge loop (see below) |
| 7 | Wiki Compliance Review | Validate plan against agreed wiki sections from Sprint 1 |
| 8 | Final Hardening | Consolidate all sections, remove placeholders, write/update ADRs, produce architecture change descriptions |

#### How the Plan Document is Written (Skeleton-First)

The plan is a **single living document** that grows incrementally across sprints:

1. **Sprint 1** creates two artifacts:
   - `wiki-scope.md` — the agreed list of applicable wiki sections and their relevance
   - `<plan-name>.md` — a **skeleton** of the full plan following the `create_plan_generic.md` structure, with every section heading present but body content as one-line stubs (e.g., `<!-- TODO: Sprint 2 will fill this section -->`)

2. **Sprints 2–7** each read the current plan via Read tool, fill in their designated section(s), and may revise earlier sections if the critic identifies inconsistencies between sections.

3. **Sprint 8** does a final read of the entire plan, removes all remaining stubs, ensures cross-section consistency, produces the architecture change descriptions and ADR entries.

The critic evaluates both the new sprint's section **and** whether it conflicts with or contradicts previously written sections.

The plan structure follows `create_plan_generic.md` exactly. The harness injects the content of that file into the generator's sprint prompt at runtime (Python-side), so updates to `create_plan_generic.md` are automatically picked up by future runs without any code changes.

#### Per-Issue Research/Challenge Loop (Sprint 6 and any sprint on escalation)

Sprint 6 uses `architecture_critique.md` as the critic framework. After the critic produces its structured feedback, for each identified issue the harness runs a **focused generator sub-round**:

> "Please examine this review feedback carefully. Research it. Look at how it matches the software-backend-wiki. Do external searches. Ensure it's valid. Examine each recommendation and challenge it. Potentially propose a better solution. It is OK to push back, or to declare this an invalid or out-of-scope concern. Ask questions if needed."

The critic then reviews the generator's response to each challenge and either:
- **Accepts** the pushback (removes the issue or downgrades severity)
- **Requires revision** (maintains the issue with refined guidance)
- **Escalates to the user** (pauses the loop and prompts Gerald directly) for issues where expert judgment is needed

This loop is scoped to Sprint 6 by default but the architecture leaves room to trigger it in other sprints if the critic flags a concern that warrants deep research.

#### Generator Access

- Read access to: `software-backend-wiki` (fetched at runtime), input documents, existing `knowledge/architecture/` files, existing `knowledge/adr/` files
- Write/Edit access to: plan document, `wiki-scope.md`, ADR files in `knowledge/adr/`
- Tools: `Read, Write, Edit, Bash, Glob, Grep` (same as architecture generator)

#### Critic Access

- Read access to: plan document, `wiki-scope.md`, `software-backend-wiki`, `architecture_critique.md`
- Sprint 6 critic uses `architecture_critique.md` as its evaluation framework
- Returns structured JSON feedback (`CriticResult` — same model as architecture critic)
- Tools: `Read, Bash, Glob, Grep` (read-only, same as architecture critic)

#### Architecture Document Updates

If `knowledge/architecture/` exists in the working directory, Sprint 8 produces a dedicated section in the plan describing **precise, file-level changes** to be made to the existing architecture documents as a result of implementation decisions made during planning. This section describes what to change and why — it does not make the changes itself (the plan is a plan, not an implementation).

Format per file:
```
### knowledge/architecture/<filename>.md
**Change**: [Describe the specific addition, revision, or new section]
**Reason**: [Why this implementation decision requires an architecture update]
```

#### ADR Handling

Key decisions made during planning are written as ADR files directly to `knowledge/adr/` using the project's existing ADR format (`ADR-NNN-kebab-case-title.md`). The generator:
- Creates **new ADR files** for decisions not yet documented
- Proposes **updates to existing ADRs** when a planning decision supersedes or refines a prior decision (following the existing pattern of marking ADRs as "Superseded by ADR-NNN")
- Accumulates ADR entries throughout the sprint run; Sprint 8 finalizes and ensures consistency

#### Quality Gates

- Same as `adversarial-architect`: ≥9.0 avg critic score, 2 consecutive passing rounds, zero Critical/High issues to exit a sprint
- Git auto-commit after each round (same pattern)
- Prompt templates as `.md` files loaded at runtime via `load_prompt()`

#### Output Folder

`knowledge/plans/<plan-name>/` containing:
- `<plan-name>.md` — the final plan document
- `wiki-scope.md` — agreed wiki sections
- Sprint contracts and critic feedback artifacts (in `.checkpoints/`)

### Nice-to-Have Features

- Team-mode critique for Sprint 6 (architect-lead + specialist experts in parallel) using the team mode defined in `architecture_critique.md`
- Configurable wiki URL/path (default: `https://github.com/geraldthewes/software-backend-wiki`)
- Summary report at end of run showing which wiki sections were applied and which ADR decisions were made

### Out of Scope

- Modifying `adversarial-architect` behavior
- Coding agent / implementation phase (plan generation only, not code generation)
- GUI or web interface
- Support for wikis other than `software-backend-wiki` (initially)
- Automatic handoff to coding agents after plan completion

## Constraints & Context

**Integration Points**:
- Shares the `deep_architect` Python package with `adversarial-architect`
- Output feeds into future coding agent workflows
- `software-backend-wiki` at `https://github.com/geraldthewes/software-backend-wiki` is authoritative best-practices source
- `architecture_critique.md` at `.claude/commands/architecture_critique.md` is the Sprint 6 critique framework; injected into critic prompt at runtime
- `create_plan_generic.md` at `.claude/commands/create_plan_generic.md` defines the plan document structure; injected into generator prompt at runtime by the harness (Python-side, so updates to the file are automatically picked up)

**Preferred Technologies**: Python, same stack as `deep-architect` (uv, Typer, Pydantic, claude-agent-sdk, ruff, mypy, bandit, pytest)

**Constraints**:
- Maximum code reuse with `adversarial-architect` — shared logic must not be duplicated
- All new code must pass ruff, mypy (strict), pytest, and bandit before committing
- No hardcoded thresholds — read from `HarnessConfig` (extend TOML config if new thresholds needed)
- No new agent frameworks — plain async Python orchestration only
- Prompt templates as `.md` files, not inline strings
- ADRs written for any new architectural decisions

## Research Questions for Planning Phase

- What is the cleanest way to share the harness loop between `adversarial-architect` and `adversarial-plan` without coupling them? (Abstract base class? Shared `harness_core.py`? Parametric `run_harness()`?)
- How should the per-issue research/challenge loop be orchestrated as sub-rounds within Sprint 6 — as a separate inner loop after the main critic round, or woven into the round structure?
- How should critic escalations to the user be implemented — pause and prompt interactively, or write an escalation report and exit cleanly?
- What is the best way to inject `software-backend-wiki` content into agents — clone the repo into a temp dir, fetch specific pages at runtime, or pre-cache relevant sections into the workspace?
- What new `SprintDefinition` fields (if any) are needed to support the wiki compliance sprint variant and the architecture critique sprint with its per-issue loop?
- Should the plan skeleton created in Sprint 1 have fixed section names derived from `create_plan_generic.md`, or should Sprint 1 be allowed to adapt the structure based on the input documents?
- How should ADR numbering be determined at runtime to avoid conflicts with existing ADRs in `knowledge/adr/`?
- Review `knowledge/adr/` for existing decisions that constrain the design (especially ADR-003 tool access, ADR-011 resume/checkpoint, ADR-021 stateless sessions, ADR-023 rollback, ADR-025 structured output)
