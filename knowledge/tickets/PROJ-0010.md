# Add support for yolo mode

> **Workflow Status**: Requirements gathered ✓ → Ready for `/create_plan_generic` to research and plan

## Problem Statement & Business Context

**What problem are we solving?**
The tool currently runs fully unattended through all 7 sprints. There is no way to review sprint output before the next sprint begins. If a problem exists at sprint 3, sprints 4–7 run on a flawed foundation, wasting time and producing an unreliable architecture document.

**Why is this important?**
Human review between sprints allows course-correction early. The architecture review process benefits from checkpoints where the user can inspect, and optionally edit, the generated files before committing to the next sprint.

**Who needs this?**
All users of `adversarial-architect` who want incremental control over the generation process rather than a fully automated run.

## Requirements & Scope

### Must-Have Features
- Add a `--yolo` CLI flag to `adversarial-architect`
- **Default behavior (no `--yolo`)**: after completing each sprint, the tool stops, prints the files the user should review, prints the command to resume, and exits
- **`--yolo` behavior**: runs all sprints unattended — identical to today's behavior
- `--resume` (without `--yolo`): resumes at sprint N+1, then stops again at the end of that sprint
- `--resume --yolo`: resumes at sprint N+1 and runs the remaining sprints unattended
- `--yolo` is a per-invocation flag — it is not persisted in saved state

### Nice-to-Have Features
- None identified

### Out of Scope
- No new abort or recovery logic beyond what already exists
- The tool does not re-run the critic after human edits; it trusts the human and moves directly to sprint N+1
- `--yolo` preference is not saved between sessions

## Constraints & Context

**Integration Points**: `cli.py` (Typer CLI), `harness.py` (sprint loop orchestration), `io/files.py` (state persistence), `models/progress.py` (`HarnessProgress`)

**Preferred Technologies**: Typer for CLI flag; no new dependencies

**Constraints**:
- The existing `--resume` mechanism (PROJ-0009) already handles loading saved state — no new resume logic is needed
- The stop point must save all state so a subsequent `--resume` picks up cleanly at sprint N+1
- The message printed at the stop point must clearly identify which files to review and the exact command to resume

## Success Criteria

### Automated Verification
- [ ] `adversarial-architect --help` shows `--yolo` flag with description
- [ ] Unit tests cover the flag being passed through to the harness
- [ ] Existing tests continue to pass

### Manual Verification
- [ ] Running without `--yolo`: tool completes sprint 1, stops, prints relevant output files and resume command, exits cleanly
- [ ] Running with `--yolo`: tool runs all sprints unattended (current behavior preserved)
- [ ] Running `--resume` (no `--yolo`): resumes at sprint N+1, stops at end of that sprint
- [ ] Running `--resume --yolo`: resumes at sprint N+1, runs remaining sprints unattended

**End State**: Users have the choice between fully automated runs (`--yolo`) and a checkpoint-based mode (default) where human review happens between every sprint.

## Research Questions for Planning Phase

The following should be investigated during `/create_plan_generic`:

- Where in `harness.py` does the sprint loop run and where is the natural stop point to insert the pause?
- How does `HarnessProgress` track the current sprint — is the completed sprint number already persisted to disk?
- How are CLI flags currently defined in `cli.py` with Typer — what's the pattern for adding `--yolo`?
- How does the harness need to communicate `--yolo` down to the sprint loop (parameter threading)?
- What files are relevant for a given sprint — how should the stop message be constructed?
