# Add sprint-boundary git commits to capture metadata and mark sprint milestones

## Problem Statement & Business Context

**What problem are we solving?**
The harness commits architecture files after each generator round (ADR-010), but no commit marks sprint completion. This leaves critic feedback, progress state, and history entries from the final round of each sprint uncommitted to git. The git history also lacks navigable sprint boundaries, making it difficult to review architectural evolution or roll back to a specific sprint's approved state.

**Why is this important?**
Three concrete gaps exist in the current commit strategy:

1. **Metadata gap** — After the final generator round of a sprint, the critic evaluates and writes feedback JSON, round logs, critic history, and updated `progress.json` to disk. No subsequent generator round exists to sweep these into git via `get_modified_files()`. For the last sprint, these artifacts are never committed at all.
2. **No sprint boundaries in git log** — `git log --oneline` shows a flat stream of `"Generator pass N - sprint M"` commits. Identifying where one sprint ends and the next begins requires cross-referencing `progress.json` or feedback files.
3. **Rollback requires detective work** — Rolling back to "the approved state after Sprint 3" means manually identifying which `"Generator pass"` commit was the last one for that sprint. A boundary commit makes this trivial.

**Who needs this?**
Anyone reviewing the architecture output, auditing the evolution of design decisions, or needing to roll back a problematic sprint.

## Requirements & Scope

### Must-Have Features

- After a sprint passes (status transitions to `"passed"`), call `get_modified_files()` + `git_commit()` to commit **all** modified and untracked files with message: `"Sprint {N} complete: {sprint_name}"`
- This commit must capture at minimum: final critic feedback JSON, updated `progress.json`, `generator-history.md`, `critic-history.md`, and any round logs from the final round
- After the final mutual agreement round completes and `progress.status = "complete"`, commit all remaining artifacts with message: `"Architecture complete — all {N} sprints passed"`
- Existing per-round generator commits (ADR-010) remain unchanged — this extends, not replaces, that behavior

### Nice-to-Have Features

- Add a lightweight git tag at each sprint boundary: `sprint-{N}-complete` (enables `git diff sprint-3-complete..sprint-4-complete`)
- Add a final tag: `architecture-complete`

### Out of Scope

- Squashing per-round commits within a sprint (separate UX decision)
- Changing the per-round commit behavior from ADR-010
- Branching strategies per sprint

## Constraints & Context

**Integration Points**: `git_ops.py` (existing `get_modified_files` + `git_commit`), `harness.py` sprint loop and final agreement block

**Constraints**:
- Must not break `--resume` flow — sprint-boundary commits are additive
- Must not alter the per-round commit messages or behavior (ADR-010)
- The sprint-boundary commit is a no-op if no files changed since the last generator commit (use existing `git_commit` no-op logic)

## Success Criteria

### Automated Verification
- [ ] After each sprint passes, a commit exists containing the sprint's final critic feedback, updated progress.json, and history files
- [ ] After the harness completes, a final commit exists with `progress.status = "complete"`
- [ ] Commit messages follow the format `"Sprint {N} complete: {sprint_name}"` and `"Architecture complete — all {N} sprints passed"`
- [ ] Per-round generator commits (ADR-010) are unaffected
- [ ] All existing tests pass; new tests verify sprint-boundary commits are created

### Manual Verification
- [ ] Run a multi-sprint harness; confirm `git log --oneline` shows clear sprint boundaries interleaved with per-round commits
- [ ] Confirm `git show` on a sprint-boundary commit includes feedback, progress, and history files
- [ ] If tags implemented: confirm `git tag -l` shows sprint milestone tags

**End State**: Git history contains explicit sprint-boundary commits that capture all metadata artifacts and provide navigable milestones for review and rollback.

## Notes

The implementation is intentionally small (~10-15 lines of production code). The sprint-boundary commit reuses the existing `get_modified_files()` + `git_commit()` pattern — no new git abstractions needed. If tags are added, a small `git_tag()` helper in `git_ops.py` would be the only new function.
