# Ensure modified code follows repo quality checks and coding standards

> **Workflow Status**: Requirements gathered ✓ → Ready for `/create_plan_generic` to research and plan

## Problem Statement & Business Context

**What problem are we solving?**
After `review-action` (part of the review-analyzer tool set) has an agent address a code-review
finding, there's currently no guarantee the fix actually conforms to the target repo's quality
standards. Different repos define different toolchains and scopes (as seen in `plant-tracking`:
root CLI, FastAPI backend, and `plant_service` package each with their own lint/format/type-check/
security/test commands), and some standards are style/convention rules that require LLM judgment
rather than a programmatic tool (e.g. `.opencodereview/rules/python-style.md`).

**Why is this important?**
The standards defined in the target repo need to actually be followed — a fix isn't really "done"
until it's been verified against those standards, the same way a human or coding-agent workflow
would validate it.

**Who needs this?**
A developer using `review-action`.

**Note**: Some checks are programmatic (run a CLI tool, parse pass/fail — e.g. ruff, mypy, pytest,
bandit). Others require LLM judgment (e.g. conformance to a style-guide markdown doc like
`.opencodereview/rules/python-style.md`). Both kinds need to be supported.

## Reference Implementation (To Research)

**Location**: `~/repos/plant-tracking`

**Aspects to mimic**:
- Its per-subproject quality-check structure:
  - Root CLI package (`pyproject.toml`, tests in `tests/`): pytest, ruff, black, mypy (permissive,
    root `mypy.ini`), bandit (excludes B403–B409), all bundled via `just check`
  - FastAPI backend (`backend/fastapi/`): pytest, ruff (line-length 100, py310), black, mypy,
    bundled via `just api-check`
  - `plant_service` package (`packages/plant_service/`): pytest, ruff (scoped to E, F, W, C90;
    C901 ignored), mypy, bandit as dev deps — no single bundled command, run ad hoc
- Its `.opencodereview/rules/` convention for LLM-judged style rules — research phase should
  inspect this directory directly in `plant-tracking` to learn the convention, since
  `opencodereview` already defines it

## Requirements & Scope

### Must-Have Features
- A standard way for a repo to declare its programmatic quality-check commands (lint/format/
  type-check/security/test)
- A standard way for a repo to declare LLM-judged style/convention rules (following the
  `.opencodereview/rules/*.md` convention)
- `review-action` runs both kinds of checks against files it just modified, after addressing a
  review finding, before considering the fix "done"
- Check failures (either kind) feed back into the review-action fix loop — retry until clean,
  with a proper exit condition so it doesn't loop forever
- The declared-checks convention should be usable both by `review-action` and by a regular coding
  agent (not a review-action-only format)

### Nice-to-Have Features
- Auto-detection of checks from existing repo config (`Justfile`, `pyproject.toml`, etc.) without
  requiring an explicit new declaration file when not needed

### Out of Scope
- Non-Python toolchains/languages (initial focus is Python)

## Constraints & Context

**Integration Points**: Lives within the review-analyzer tool set, integrating with
`deep_architect/review_action_harness.py`'s existing fix loop.

**Preferred Technologies**: None mandated — pick what fits (may reuse patterns from
`exit_criteria.py`/ping-pong logic already in this repo).

**Constraints**:
- No hard performance/security constraints specified
- Default to fail-closed behavior: a finding is never marked resolved while checks fail

## Success Criteria

### Automated Verification
- [ ] `review-action`, after addressing a finding, automatically discovers and runs the target
      repo's programmatic checks (lint/format/type-check/security/test) plus any LLM-judged style
      rules
- [ ] Failing checks trigger a retry of the fix; a finding is never marked resolved while checks
      fail
- [ ] The retry loop has a bounded exit condition (doesn't run forever)
- [ ] Validated end-to-end against `plant-tracking` (root CLI, FastAPI backend, `plant_service` —
      all three check profiles)
- [ ] `.opencodereview/rules/python-style.md`-style rules are enforced via an LLM judgment step
- [ ] Dogfoods cleanly against `deep-architect`'s own checks (ruff/mypy/pytest/bandit)

### Manual Verification
- [ ] Manually run `review-action` against `plant-tracking` and observe the check loop behaving
      correctly across all three subproject profiles

**End State**: `review-action` automatically discovers and runs a target repo's real quality
gates (programmatic + LLM-judged) after each fix attempt, looping until clean or a bounded exit
condition is hit, before ever marking a finding resolved.

## Research Questions for Planning Phase

The following should be investigated during `/create_plan_generic`:
- How does `.opencodereview/rules/` in `plant-tracking` structure and expose its convention (file
  format, discovery mechanism, how `opencodereview` itself consumes it)?
- What does `deep_architect/review_action_harness.py` currently do after a fix is applied — where
  does the new check-loop hook in?
- How should a repo declare its programmatic check commands (per-subproject, like
  `plant-tracking`'s root/backend/`plant_service` split) so it can be auto-detected — is there a
  common signal to sniff (`Justfile`, `pyproject.toml` scripts, `pre-commit` config) before
  falling back to an explicit declared-checks file?
- How should the loop's exit criteria/max-retries be modeled — reuse patterns from
  `exit_criteria.py` and the ping-pong logic, or does this need its own?
- How does an LLM-judged check (style-guide conformance) get invoked — a dedicated agent call per
  rule file? What does its structured output/pass-fail contract look like?
- What's the right shared format so both a coding agent (e.g. via CLAUDE.md conventions) and
  `review-action` can use the same declared-checks convention?
